# Improved DefectDojo-like ASOC Platform

Масштабируемая платформа управления уязвимостями  
(улучшенный аналог DefectDojo)

---

## 📌 Описание

Этот репозиторий описывает архитектуру и дизайн **масштабируемой ASOC / vulnerability management платформы**, вдохновлённой DefectDojo, но спроектированной с учётом:

- обработки **10⁸–10⁹ уязвимостей**
- большого количества источников данных
- высокой нагрузки на дедупликацию и корреляцию
- необходимости простой установки (on-prem / cloud)
- внедрения **ИИ как помощника**, а не “чёрного ящика”

Цель проекта — предложить **future-proof архитектуру**, подходящую как для внутреннего использования в крупных компаниях, так и для построения собственного локального ASOC.

---

## 🎯 Цели проекта

- Масштабируемая дедупликация уязвимостей при огромных потоках данных
- Простая и гибкая архитектура (Docker / Helm)
- API-first подход и лёгкая интеграция сканеров
- Поддержка стандартных форматов (SARIF)
- Возможность расширения без переписывания ядра
- Готовность к multi-tenancy
- Интеллектуальные функции (AI/ML) с контролем человеком

## 🧠 Ключевые идеи

### 1. Масштабируемая дедупликация

Дедупликация рассматривается как **ключевая проблема производительности**, а не вспомогательная функция.

**Базовый подход**
- Для каждой находки вычисляется **fingerprint (хеш)** из нормализованных полей  
  (тип уязвимости, CVE/CWE, asset, путь, порт и т.д.)
- В базе данных используется **UNIQUE индекс по fingerprint**
- Проверка дублей переносится на уровень БД → O(1) операция

**Дополнительно**
- Вторичный хеш при редких коллизиях
- Многоуровневая дедупликация:
  - внутри одного скана
  - между сканами одного приложения
  - глобально между разными инструментами
- Асинхронная обработка через очередь
- Хранение истории дублей (дубликаты помечаются, но не теряются)

---

### 2. Архитектура

Архитектура проектируется как **модульная и масштабируемая**, с возможностью как простого, так и распределённого деплоя.

**Основные компоненты**
- **Ingest Service** — приём данных от сканеров (API, webhooks)
- **Processing / Deduplication Service** — дедупликация, корреляция, правила
- **API & UI Service** — REST API и веб-интерфейс
- **Integrations Service** — Jira, webhooks, уведомления
- **AI / ML Service (опционально)** — triage, enrichment, отчёты

**Принципы**
- Stateless-сервисы
- Асинхронная обработка
- Idempotency для всех критичных операций
- Возможность “all-in-one” деплоя для малого масштаба

---

### 3. Интеграции

Платформа строится как **integration hub**, а не просто “загрузчик отчётов”.

**Поддерживается**
- REST API для импорта находок
- Webhooks для входящих и исходящих событий
- Плагинная система парсеров
- SARIF как универсальный формат
- Возможность подключения кастомных сканеров без форка ядра

**Событийная модель**
- NewFinding
- FindingStatusChanged
- SLA_Breached
- AutoTriageSuggested

---

### 4. Технический стек (рекомендация)

| Компонент | Выбор | Обоснование |
|---------|------|------------|
| Backend | Python + Django / DRF | Экосистема AppSec, быстрый старт, ML |
| БД | PostgreSQL | Партиции, индексы, транзакционность |
| Очередь | RabbitMQ + Celery | Надёжная асинхронность |
| Кэш | Redis | Кэш, locks, rate limiting |
| Frontend | React| Гибкость UI |
| Контейнеризация | Docker / Helm | Простота деплоя |

---

### 5. ИИ / ML (Assist, not Replace)

ИИ используется **как помощник**, а не как автоматическое принятие решений.

**Применения**
- Вероятностная дедупликация между разными сканерами
- Авто-приоритезация (CVSS + EPSS + контекст)
- Предсказание False Positives
- Классификация по CWE / OWASP / ATT&CK
- Генерация отчётов и пояснений (LLM)

**Принципы**
- Всегда есть human-in-the-loop
- Возможность отключить AI-модули
- Прозрачность решений модели
- Обучение на действиях пользователя

---

### 6. Масштабирование и отказоустойчивость

- Горизонтальное масштабирование сервисов
- Репликация БД (read replicas)
- Партиционирование таблиц
- Backpressure и rate limiting
- Circuit Breaker для внешних интеграций
- Kubernetes + HPA
- Метрики и observability (Prometheus)

---

### 7. Multi-tenancy и расширяемость

**Поддерживаемые модели**
- Single DB + tenant_id
- Schema per tenant
- Separate DB per tenant (для крупных клиентов)

**Future-proof подход**
- Версионирование API
- Плагинная архитектура
- Минимум hard-coded логики
- Расширение через API и события

---

## 🚫 Non-Goals (на старте)

- Полная оркестрация сканеров
- Автоматическое закрытие уязвимостей без человека
- Зависимость от конкретного облака
- Монолит без возможности разделения
